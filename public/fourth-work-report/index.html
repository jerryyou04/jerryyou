Abstract / Introduction

For my Winter 2026 co-op term, I worked as a Software Developer Intern at TD Bank on the Enterprise Governance, Risk, and Compliance (eGRC) team.

Most of my work sat in a part of software that is rarely highlighted in school: testing existing logic, debugging edge cases, moving data reliably, and building reporting systems that make internal activity visible.

The main takeaway from this term is that real-world systems are not clean or ideal. The valuable skill is learning how to work within that reality: understanding messy systems, validating assumptions, and improving things without breaking everything else.

About the Employer

TD Bank is one of Canada's largest financial institutions, with operations across banking, investing, insurance, and internal technology systems. During the term, I worked from TD Terrace in Toronto.

My role focused on internal software rather than customer-facing products. Specifically, I worked within an eGRC environment where systems support governance workflows, compliance tracking, and reporting. In this context, even small logic issues can have real downstream effects, which made reliability and testing especially important.

Learning Goals

Goal 1 - Technological Literacy

Goal: Develop hands-on experience with enterprise-level systems and internal tools.

Reflection: This goal was strongly achieved. I worked extensively with ServiceNow, REST APIs, SQL Server, and Power BI, but more importantly, I learned how these tools actually interact in a production environment.

Unlike school projects where systems are built from scratch and are relatively clean, I worked with existing codebases that required understanding assumptions, dependencies, and historical decisions. Writing Jest tests for ServiceNow scripts forced me to break down logic into testable pieces and simulate platform behavior using mocks, such as GlideRecord.

I also contributed to a data pipeline that extracted ServiceNow data into SQL Server and fed Power BI dashboards. Overall, I did not master every tool, but I became significantly better at navigating unfamiliar systems and figuring out how things actually work.

Goal 2 - Oral Communication

Goal: Improve professional communication in a technical environment.

Reflection: This goal was moderately achieved. I participated in team meetings, provided updates on progress, and documented my work through tickets and internal notes.

The main challenge was learning how to communicate clearly when the system itself was unclear. Explaining bugs, edge cases, or unexpected behavior required not just describing what happened, but also what assumptions might be wrong.

Over time, I improved at giving concise updates and focusing on what actually mattered: blockers, risks, and next steps, instead of over-explaining technical details. While I still have room to improve, I became more comfortable communicating in a professional development environment.

Goal 3 - Personal Learning Goal: Time Management and Accountability

Goal: Improve independence, time management, and ownership of tasks.

Reflection: This goal was achieved. Much of my work required independently exploring unfamiliar code, debugging issues, and writing tests without step-by-step instructions.

To manage this, I broke tasks into smaller steps, tested assumptions early, and tracked progress through tickets. One of the biggest improvements was learning when to stop guessing and instead validate something with a test or data check.

I consistently met deadlines and required minimal revisions after code review, which indicates that I became more reliable in both execution and decision-making. This experience helped reinforce the importance of accountability in a real-world engineering setting.

Job Description

My work focused on testing infrastructure, debugging legacy logic, and building data pipelines and dashboards.

Testing ServiceNow Logic

I wrote Jest tests for ServiceNow scripts and business rules. Since ServiceNow does not run natively in a local environment, I worked with mocked implementations of platform APIs such as GlideRecord and gs. This allowed for repeatable testing and better validation of logic.

Debugging Legacy Scripts

A large part of the work involved identifying fragile logic, such as type coercion issues, incorrect assumptions about data structure, and edge cases that were not originally considered. This required careful reading, testing, and verification.

Data Pipeline and Reporting

I worked with REST APIs to extract ServiceNow data, store it in SQL Server, and build Power BI dashboards. One focus was tracking developer activity through update set metrics, which helped provide visibility into internal engineering work.

Key Technologies

  • Testing: Jest, mocked ServiceNow APIs.
  • Data: REST APIs, SQL Server.
  • Reporting: Power BI.

Reflection

The most important lesson from this term is that real-world software is uneven. Some parts are well-designed and stable, while others rely on assumptions that are not immediately visible.

Technical debt is not always obvious. It often appears as small inconsistencies: a field that is usually one type but not always, or logic that only works because of specific input patterns. Companies operate within these constraints because rewriting everything is not practical.

This changed how I approach engineering. Instead of assuming code is correct, I learned to verify behavior through testing and data inspection. I also became more comfortable working in systems where documentation is incomplete and clarity has to be built manually.

One of the more valuable realizations was that "working" code is not the same as correct code. That distinction becomes important in environments where small errors can affect reporting or workflows without being immediately visible.

Overall, I improved my ability to reason through unclear systems, debug effectively, and make incremental improvements in a complex environment.

Conclusion

This work term was not just about learning specific technologies like Jest, ServiceNow, SQL Server, or Power BI. It was about understanding how software behaves inside a large organization.

If someone were to summarize this experience, I would want them to say that I learned how to work with real systems, not ideal ones, and how to make them slightly more reliable and understandable.

I left the term more confident in my technical ability, but also with a more realistic view of how software development works in practice.
